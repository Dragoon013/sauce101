    Sauce Labs History

  • Founded in 2008
  • Steven Hazel, John Dunham, and Jason Huggins (Co-creator of Selenium)

  • Developed with the purpose of providing:
    • Parallel Testing
    • Test Variability

    • Sauce Labs was founded in 2008 by Jason Huggins, co-creator of Selenium (the open source standard for automated testing frameworks) and Steve Hazel, current CTO. They set out to create a cloud-based platform for automated testing to free development teams from the hassle and cost of testing so they can focus on innovating their core capabilities.

    Jason was using Selenium to do browser testing, and the process of testing the apps in different browsers was taking too long. The problem was that all the testing was being done on one machine, so he split the tests onto 4 different machines, effectively running the tests in parallel, which cut the time drastically.

    • Cogs in the Selenium Machine

    There are actually a lot of parts that make up Selenium!

  • Selenium IDE - A firefox add-on that can create a script from a manual test, allow you to do maual testing, and debug your own test scripts.
  • Selenium Client API - what we will be using today, as part of the Selenium Webdriver. It allows tests to be written in a variety of programming languages, so it enables us to write our tests in Java
  • Selenium WebDriver - The Selenium WebDriver is the client that accepts the commands via the API or in the Selenium scripting language and sends those commands to a browser.
  • Selenium Grid - this is the Selenium Server, which allows tests to to use web browsers on other machines - so this is what Sauce Labs does for you, but on a very large scale and with a lot of options for machines and browsers - not restricted to just one version either

    • How Selenium WebDriver Works

    Local:

    1. Write script
    2. Run script (send commands via API)
    3. WebDriver Client sends commands to browser
    4. WebDriver Client sends results back

    Sauce Labs:

    1. Write script
    2. Run script (send commands via API)
    3. WebDriver Client sends commands to browser on remote Selenium Grid (Sauce Labs)
    4. WebDriver Client sends results back
    5. Test Assets Created

    Anticipate Browser Response

  • Implicit Waits
    •                         
                          driver.manage().timeouts().implicitlyWait(10, TimeUnit.SECONDS);
  • Explicit Waits
    •                         
                          import org.openqa.selenium.support.ui.ExpectedConditions;
import org.openqa.selenium.support.ui.WebDriverWait;
WebDriverWait wait = new WebDriverWait(driver, 10);
WebElement messageElement = wait.until( ExpectedConditions.presenceOfElementLocated(By.id("loginResponse")) );

    When you're performing an action, it takes time for the action to reach the server and then time for the server to respond. If you are trying to test something that depends on the previous action or response, then you need to build that wait time into your test. Otherwise, the test might fail because the elements that are expected for the next step haven't loaded into the browser you.

    While we may not have waits in our current script, we will take a look at them in the next module.
